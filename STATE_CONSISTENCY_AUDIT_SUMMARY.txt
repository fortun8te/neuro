BUGS FOUND AND FIXED
================================================================================

1. TASK SCHEDULER RACE CONDITION (AppShell.tsx)
   Severity: HIGH
   Status: FIXED ✓
   Issue: Task marked RUNNING, navigation attempted, completion marked after 2s
   Causes: Race condition between state update and navigation, timeout can fail
   Fix: Store previous state, add rollback on navigation failure

2. CASCADE DELETE PARTIAL UPDATE (storage.ts)
   Severity: CRITICAL
   Status: FIXED ✓
   Issue: Multi-step delete (cycles → images → campaign) can leave orphaned data
   Causes: No rollback on partial failure, in-place mutations
   Fix: Pre-flight validation, per-phase rollback logic

3. ORPHANED CHECKPOINT (sessionCheckpoint.ts)
   Severity: HIGH
   Status: FIXED ✓
   Issue: Checkpoint saved but session list not updated (two separate writes)
   Causes: Non-atomic multi-step mutation, in-place modifications
   Fix: Atomic write using Promise.all(), calculate state before mutation

4. REPEAT TASK RESCHEDULE (taskScheduler.ts)
   Severity: MEDIUM
   Status: FIXED ✓
   Issue: Failed reschedule leaves task in PENDING, loops endlessly
   Causes: No error state for reschedule failure, event fires without DB update
   Fix: Mark FAILED on reschedule failure, immutable state updates

5. CONCURRENT METRICS SAVE RACE (storageAdapter.ts)
   Severity: MEDIUM
   Status: FIXED ✓
   Issue: Two concurrent saveMetrics calls can lose one call's updates
   Causes: Missing write queue serialization pattern
   Fix: Add saveMetricsQueue (same pattern as storage.ts _cycleWriteQueue)

6. ERROR STATE STALENESS (useStorage.ts)
   Severity: LOW
   Status: FIXED ✓
   Issue: Old error message persists after successful retry
   Causes: Error cleared only on success, not on operation start
   Fix: Clear error at START, add isLoading state for operation progress

KEY PATTERNS APPLIED
================================================================================

PATTERN 1: Previous State Snapshots
  - Store state before multi-step operations
  - Use as rollback point on failure
  - Example: backup cycles before deleting

PATTERN 2: Atomic Writes with Promise.all()
  - All database updates together
  - Prevents orphaned references
  - Used in: sessionCheckpoint save, checkpoint pruning

PATTERN 3: Write Queue Serialization
  - Queue all writes to prevent concurrent races
  - Pattern: _queue = _queue.catch(() => {}).then(op)
  - Used in: storage.ts (_cycleWriteQueue), sessionCheckpoint.ts (_checkpointWriteQueue)
  - NOW ADDED: storageAdapter.ts (saveMetricsQueue)

PATTERN 4: Phase-Based Multi-Step Operations
  - Phase 1: Validate & Backup
  - Phase 2: Delete with rollback
  - Phase 3: Next delete with previous rollback
  - Example: deleteCampaign (cycles → images → campaign)

PATTERN 5: Explicit Error States
  - Don't leave tasks "pending" on error
  - Use specific error states: "failed" with error message
  - Prevents silent failures and infinite retries

PATTERN 6: Clear State on Operation Start
  - Reset error/loading at beginning
  - Don't wait for success to clear old state
  - Prevents stale error messages on retry

================================================================================
COMMON ROOT CAUSES IDENTIFIED
================================================================================

1. SEQUENTIAL WRITES WITHOUT ATOMICITY
   - Multiple await/set calls in sequence
   - First succeeds, second fails → inconsistent state
   - Fix: Use Promise.all() for atomic writes

2. IN-PLACE MUTATIONS WITH RETRIES
   - Object mutations in retry loop
   - Rollback impossible after mutations
   - Fix: Calculate new state, mutate once at end

3. MISSING WRITE SERIALIZATION
   - Concurrent operations to same resource
   - Lost updates from interleaved writes
   - Fix: Add write queue pattern

4. NO ROLLBACK ON PARTIAL OPERATIONS
   - Multi-step operations lack rollback logic
   - Cleanup code doesn't restore previous state
   - Fix: Store previous state, call restore in catch

5. TIMEOUT-BASED STATE UPDATES
   - setTimeout for marking completion
   - Can fail, leaving task in wrong state indefinitely
   - Fix: Synchronous updates when operations complete

6. STATELESS ERROR HANDLING
   - Error state not cleared on retry
   - isLoading flag missing
   - Fix: Clear state at operation START
