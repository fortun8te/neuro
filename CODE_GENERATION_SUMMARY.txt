ARCHITECTURE DECISIONS
─────────────────────────────────────────────────────────────────────────────

Why separate CodeQualityValidator?
  - Can validate any code before output
  - Reusable across stages and contexts
  - Extensible for future languages

Why artifact types (visualization, component, etc.)?
  - Provides context for generation
  - Enables language-specific templates
  - Supports diverse business use cases

Why ContextAwareGenerator?
  - Prevents duplicate code
  - Matches team conventions
  - Suggests correct integration points
  - Reduces friction when using generated code

Why makeStageCodeGeneration separate?
  - Business logic isolated from engine
  - Easy to swap for other stages
  - Can update Make strategy independently

KNOWN LIMITATIONS
─────────────────────────────────────────────────────────────────────────────

Current
  - Validation is heuristic-based (not full compiler)
  - No execution sandbox (assumes valid)
  - Limited to 5 languages (TS, Python, SQL, Markdown, JavaScript)
  - Basic duplicate detection (string matching only)

Not Supported
  - C++, Go, Rust, Java, Kotlin (future milestone)
  - Interactive code generation (static only)
  - Code migration (only new code)
  - Visual components (use Figma MCP for that)

Mitigations
  - Validation catches most issues before output
  - All generated code includes error handling
  - Tests recommended for production use
  - Optional artifact review gate available
